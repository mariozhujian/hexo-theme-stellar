秒杀系统的本质不是“让所有请求都进入订单系统”，而是用尽可能低的成本，快速筛选出少量真正有资格购买的请求。

整体链路可以概括为：

```text
用户
 → CDN/静态页面
 → API 网关（限流、防刷）
 → 秒杀服务（资格校验、Redis 扣库存）
 → 消息队列
 → 订单服务（异步创建订单）
 → 数据库
```

## 1. 它解决什么问题？

秒杀系统主要解决“极短时间内，超大流量争抢极少库存”的问题。

核心目标包括：

- 防止库存超卖。
- 承受瞬时高并发，避免数据库被打垮。
- 保证同一用户不能重复购买。
- 尽量公平，先到先得。
- 快速返回结果，避免用户长时间等待。
- 隔离秒杀流量，避免拖垮普通商城业务。
- 应对脚本、机器人、恶意刷接口。

例如：

```text
库存：1,000 件
瞬时请求：100 万
真正需要进入订单系统的请求：最多约 1,000 个
```

因此，秒杀系统最重要的设计思想是：

> 在流量到达数据库之前，逐层拦截和削峰。

## 2. 底层原理是什么？

### 2.1 页面静态化

商品详情、活动规则等静态资源放到 CDN，避免每次请求都访问后端。

动态信息只保留：

- 是否开始。
- 剩余状态。
- 用户是否有资格。
- 秒杀接口地址或令牌。

### 2.2 分层限流

通常在多个层次限流：

```text
CDN/WAF → Nginx → 网关 → 秒杀服务 → 用户维度
```

常见算法：

- 固定窗口。
- 滑动窗口。
- 漏桶。
- 令牌桶。

Java 项目常用：

- Sentinel。
- Redis + Lua。
- Guava RateLimiter。
- 网关自带限流插件。

### 2.3 Redis 预扣库存

活动开始前，把数据库库存预热到 Redis：

```text
seckill:stock:商品ID → 1000
```

请求到达后，在 Redis 中原子判断并扣减库存。

不能直接使用：

```java
if (stock > 0) {
    stock--;
}
```

因为判断和扣减不是原子操作，会发生超卖。

一般使用 Lua 脚本，把以下操作放在一次原子执行中：

1. 判断用户是否已购买。
2. 判断库存是否大于零。
3. 扣减库存。
4. 记录用户购买资格。

示例：

```lua
if redis.call('sismember', KEYS[2], ARGV[1]) == 1 then
    return -2
end

local stock = tonumber(redis.call('get', KEYS[1]))
if not stock or stock <= 0 then
    return -1
end

redis.call('decr', KEYS[1])
redis.call('sadd', KEYS[2], ARGV[1])
return 1
```

### 2.4 消息队列削峰

Redis 扣库存成功，不立即同步创建订单，而是发送消息：

```text
扣库存成功 → Kafka/RocketMQ → 异步创建订单
```

消息队列的作用：

- 削峰填谷。
- 解耦秒杀服务和订单服务。
- 控制数据库写入速度。
- 支持失败重试。

用户得到的不是立即下单成功，而是：

```text
“排队中，请稍后查询结果”
```

### 2.5 数据库最终兜底

数据库仍然要做最后防线：

```sql
UPDATE seckill_stock
SET stock = stock - 1
WHERE product_id = ?
  AND stock > 0;
```

根据影响行数判断是否成功。

同时增加唯一索引：

```sql
UNIQUE KEY uk_user_activity(user_id, activity_id)
```

防止同一用户重复下单。

### 2.6 最终一致性

Redis、消息队列和数据库无法天然保持强一致，通常采用最终一致性：

- Redis 负责高速资格判断。
- MQ 负责可靠传递。
- 数据库保存最终业务结果。
- 定时任务负责对账补偿。

## 3. 常见问题是什么？

### 超卖

原因：

- 非原子扣库存。
- Redis 与数据库库存不一致。
- 消息重复消费。
- 多实例并发更新库存。

解决：

- Redis Lua 原子操作。
- 数据库条件更新。
- 用户订单唯一索引。

### 少卖

Redis 扣减成功，但消息发送失败或订单创建失败，导致库存被占用却没有订单。

解决：

- 事务消息。
- 本地消息表。
- 库存回补。
- 定时对账。

### 重复下单

可能由用户重复点击、客户端重试或 MQ 重复投递造成。

解决：

- 前端按钮防重复点击。
- Redis 用户购买标记。
- 业务幂等键。
- 数据库唯一索引。

### 热点 Key

所有请求都访问同一个商品库存 Key，可能压垮单个 Redis 节点。

解决：

- 本地库存标记。
- Redis 集群。
- 库存分片。
- 请求合并。
- 热点数据多级缓存。

库存分片需要注意：分片能够提高吞吐量，但会增加库存汇总和均衡复杂度，不能为了“架构高级”盲目使用。

### 缓存与数据库不一致

常见情况：

- Redis 已扣，数据库失败。
- 数据库成功，Redis 更新失败。
- 活动结束后库存数据不一致。

解决：

- 以数据库订单作为最终事实。
- 可靠消息。
- 补偿任务。
- 活动结束后对账。

### 消息丢失或重复

MQ 通常保证“至少投递一次”，所以重复消费很常见。

解决：

```text
消息唯一ID + 消费记录/业务唯一索引
```

消费者必须幂等。

### Redis 宕机

解决方向：

- Redis 主从和哨兵/Cluster。
- 秒杀服务快速失败。
- 本地售罄标记。
- 降级关闭活动。
- 不轻易回退到数据库扣库存，否则可能直接打垮数据库。

### 恶意刷单

解决：

- 登录校验。
- 验证码。
- 动态秒杀地址。
- 一次性令牌。
- 用户/IP/设备限流。
- 风控黑名单。
- 请求签名和时间戳。

## 4. 常见方案是什么？

### 方案一：数据库悲观锁

```sql
SELECT stock
FROM product
WHERE id = ?
FOR UPDATE;
```

优点：

- 实现简单。
- 一致性强。

缺点：

- 锁竞争严重。
- 吞吐量低。

适合流量较小的活动，不适合真正的大规模秒杀。

### 方案二：数据库乐观更新

```sql
UPDATE product
SET stock = stock - 1
WHERE id = ?
  AND stock > 0;
```

优点：

- 简单。
- 不会超卖。
- 没有显式悲观锁。

缺点：

- 大量失败请求仍会打到数据库。
- 热点行竞争严重。

适合中小规模秒杀。

### 方案三：Redis 原子扣库存

优点：

- 性能高。
- 可以拦截绝大部分无效请求。

缺点：

- Redis 与数据库存在一致性问题。
- 需要设计补偿机制。

适合高并发场景。

### 方案四：Redis + MQ + 数据库

这是互联网商城最常见的完整方案：

```text
限流
 → Redis Lua 扣库存
 → MQ 异步排队
 → 数据库创建订单
 → 对账补偿
```

优点：

- 高吞吐。
- 能削峰。
- 秒杀和订单解耦。

缺点：

- 系统复杂。
- 需要处理幂等、消息可靠性和最终一致性。

## 5. 项目里怎么用？

可以按下面的业务流程落地。

### 活动准备阶段

运营创建秒杀活动：

```text
活动ID
商品ID
秒杀价格
秒杀库存
开始时间
结束时间
用户限购数量
```

活动开始前：

1. 校验数据库库存。
2. 将秒杀库存预热到 Redis。
3. 预热商品和活动信息。
4. 生成动态接口令牌。
5. CDN 预热静态页面。

### 用户请求阶段

网关完成：

- 登录鉴权。
- 用户、IP、设备限流。
- 验证码或行为验证。
- 请求签名验证。
- 活动时间校验。

### Redis 资格判断

秒杀服务通过 Lua 原子完成：

```text
重复购买校验
+ 库存判断
+ 库存扣减
+ 用户资格占位
```

失败立即返回：

```text
已购买 / 已售罄 / 请求过于频繁
```

成功则生成业务请求 ID，写入 MQ。

### 异步创建订单

消费者执行：

1. 根据请求 ID 做幂等检查。
2. 创建秒杀订单。
3. 扣减或确认数据库库存。
4. 创建支付记录。
5. 更新秒杀结果状态。

### 查询结果

客户端轮询：

```http
GET /seckill/result/{requestId}
```

可能返回：

```text
PROCESSING
SUCCESS
SOLD_OUT
FAILED
```

也可以使用 WebSocket 或 SSE 推送结果，但多数项目使用短轮询已经足够。

### 超时未支付

订单创建后锁定支付时间，例如 10 分钟：

```text
延迟消息到期
 → 检查支付状态
 → 关闭订单
 → 回补库存
```

库存回补必须保证幂等，避免重复增加。

### Java 技术栈示例

```text
Spring Boot
Spring Cloud Gateway
Redis + Lua
RocketMQ/Kafka
MySQL
Sentinel
XXL-JOB
Prometheus + Grafana
```

关键表建议包含：

```text
seckill_activity     秒杀活动
seckill_stock        秒杀库存
seckill_order        秒杀订单
message_record       可靠消息/消费记录
stock_compensation   库存补偿记录
```

必须监控：

- 请求量和拒绝量。
- Redis 扣库存成功数。
- MQ 堆积量。
- 订单创建成功率。
- Redis 与数据库库存差异。
- 重复消费次数。
- 接口延迟和错误率。

## 6. 面试官可能追问什么？

### Redis 扣成功，MQ 发送失败怎么办？

可以采用：

- RocketMQ 事务消息。
- 本地消息表。
- Redis Stream 作为待处理队列。
- 失败回滚或定时补偿。

重点不是声称“绝对不失败”，而是保证问题可以被发现、重试和补偿。

### 为什么数据库还要扣库存？

Redis 是快速入口，数据库才是最终持久化事实。Redis 可能丢数据、故障或与数据库不一致，因此数据库需要条件更新兜底。

### Redis Lua 就能保证绝对不超卖吗？

只能保证 Redis 内部操作的原子性。整个系统仍然涉及 MQ、数据库和订单，所以数据库条件更新、唯一索引和幂等消费仍然必不可少。

### MQ 重复消费怎么办？

消费逻辑必须幂等：

```text
请求ID唯一约束
+ 用户活动唯一约束
+ 消费状态检查
```

不要依赖 MQ“绝不重复”。

### 如何保证一人一单？

至少三层保护：

```text
Redis 用户标记
+ 服务端幂等判断
+ 数据库唯一索引
```

数据库唯一索引是最终兜底。

### Redis 挂了怎么办？

秒杀业务通常选择快速失败或关闭活动，不能直接把流量全部放到数据库。

### 库存应该在什么时候扣？

通常分为：

- Redis 预扣：获得购买资格。
- 数据库确认：创建订单。
- 支付确认：真正完成销售。
- 超时关闭：释放或回补库存。

具体采用“下单减库存”还是“支付减库存”，取决于业务对超卖、少卖和用户体验的取舍。秒杀通常采用下单预占库存。

### 如何保证公平？

工程上只能做到相对公平：

- 统一服务端时间。
- 活动前不开放真实接口。
- MQ 尽量按到达顺序处理。
- 限制单用户请求频率。
- 防机器人和网络重放。

分布式系统很难保证严格的全局先后顺序。

### 库存分片怎么做？

例如把 10,000 件库存拆成 10 个 Redis Key，每个 1,000 件，请求按用户 ID 哈希到分片。但需要处理某些分片售罄、其他分片仍有库存的问题，可以采用随机重试或动态调配。

### 如何做容量评估？

重点考虑：

```text
峰值 QPS
平均响应时间
Redis 单节点吞吐量
MQ 写入及消费速度
数据库订单写入能力
可接受的排队时间
```

例如峰值 10 万 QPS、库存 1,000 件，核心不是让数据库处理 10 万 QPS，而是让限流和 Redis 拦截绝大部分请求，数据库最终只处理接近库存数量的订单。

一句话总结：

> 秒杀系统依靠静态化、限流、防刷、Redis 原子预扣、MQ 削峰、数据库兜底、幂等与补偿，把百万级争抢请求收敛成库存规模的有效订单。
